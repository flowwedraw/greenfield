Using Custom Solana RPCs with Helius and Backpack wallet

Easily switch your wallet to use custom Solana RPCs such as with Devnet or Mainnet to improve speed and reliability.

Chris St. John
11 min read
·
Apr 15, 2025

We’re going to explore using Solana Custom RPCs in Backpack wallet.

An RPC is how your wallet or app talks to the Solana blockchain. Solana has default public RPCs. The default is public and shared by everyone, which can make it slightly slower and potentially less reliable for devnet and dev work. We’ll use Helius, but you can use other providers. Backpack wallet is one of the most popular Solana wallets, created by longtime Solana dev Armani Ferrante’s Coral. Directions for both of these below.

If you normally use another wallet like Phantom, that’s okay. You can use Backpack too; it’s free to install. Even if you have a wallet already, it’s a good idea to have a different wallet for dev work anyway! Backpack makes it easy to switch RPC URLs.

This article is mainly geared toward devs, and I’ll talk a lot about Solana Devnet.

However, to be clear, custom RPC URLs are also used with Solana Mainnet for serious apps and defi traders, for example. This gives them an edge. This is for both development and live production, depending on your needs.

Why do we use a custom RPC? A custom RPC is like a specialized API mainlined into Solana blockchain. Instead of calling the default Solana RPC (API), you use a custom one that is built for speed and a lot of extra features.

Helius is a well-known RPC provider in the Solana ecosystem, but there are others like QuickNode, Triton and others, if you want to try those. They each will similarly provide you a url with an API key, as described in this tutorial.

So why not try it? It’s FREE too!

I wrote an in-depth article about Helius:
Building on Solana: Helius RPCs + Tutorial (Solana Development Platform)

Developers and serious apps usually use custom RPCs because:

  • Faster response times. Custom RPCs like Helius are optimized for speed, so transactions confirm faster and endpoints respond quicker, especially when devnet gets overloaded.
  • More reliable access. Public devnet RPCs can rate-limit or go down during high traffic. Custom RPCs reduce timeout errors and improve uptime for testing.
  • Freeload. It’s FREE to signup and you can use a pretty high amount of requests before needing to pay. Way more than you need for most dev work. Even many apps with a decent amount of users in testing can use it for free for a while without needing to pay!
  • Advanced debugging tools. Helius provides extra features like transaction simulation, logs, and parsed account data that help developers debug issues quickly.
  • Custom APIs and webhooks. Helius offers extra endpoints and webhook support so developers can get notified in real-time when transactions or programs fire off.
  • Better rate limits. Custom RPC providers usually give higher request quotas, so you don’t get blocked or limited when sending a lot of devnet transactions.
  • Works well with Backpack. Backpack makes it easy to switch networks and use custom RPCs, which pairs well with dev-focused features from providers like Helius.

If you have a wallet on filesystem (path/to/file>/wallet-dev.json) already, you can airdrop using you filesystem wallet to another address. Or, even airdrop to your filesystem wallet and then transfer from it to your browser wallet.

## assuming your have a filesystem wallet, if not, make one, 
## see in the CLI articlefor more info

solana config set --keypair </path/to/file>/wallet-dev.json
solana config set --url (your RPC url/key, OR https://api.devnet.solana.com)

solana config get

solana address

# Before airdrop
$ solana balance
#output: 0 SOL

solana airdrop 1 <leave this out or put a different pubkey>

# If not ok -- Error output - if you get a rate limit error
#output: Error: airdrop request failed. This can happen when the rate limit is reached


# After airdrop if successful
$ solana balance
#output: 1SOL

#  Or TRANSFER from you local filesystem wallet to you Backpack wallet
# this uses the keypair/wallet from the filesystem set above

solana transfer <PUBLIC_ADDRESS> 1 --no-wait

Changing RPCs in Backpack wallet

What I am going to show you is the Helius RPC for Devnet, however keep in mind you can use other providers like QuickNode, Triton and others and mainnnet or devnet versions of those.

First thing is go to your wallet, and click the upper left hand corner, mine is marked for C for the first letter of my name, yours is probably different.

btw, this wallet is called “USDC-test” because I was using it originally for that! But yours is whatever you call it.

At this point, I am on the mainnet, default settings of the wallet.

When I clicked on that icon in the left corner, I get a small popup.

Now, you should click Settings.

That brings you here:

Now you should click on “Solana”

Now you will see some settings like these:

You want to click on “RPC Connection”:

As you can see, as mentioned earlier, we are on the “default RPC” still but we will be changing that.

We want to change that to “Custom”, so click that

This shows “Recent RPCs” list. I have one listed because I was using it before, yours should be blank. If something weird is there I would delete it and start fresh.

Now add in your Helius or other RPC provider devnet url. It’s their devnet base url + your api key as it’s shown in Helius dashboard. Again, if you are new to this read my article:
Building on Solana: Helius RPCs + Tutorial (Solana Development Platform)

Hit Update and you will see this:

When I go back to my wallet I see this now with my Devnet SOL

See Helius RPC is prominently displayed, to remind you that’s what you are using (devnet if you want to use the devnet url).

If you want to remove that and be back on the default you can click the trash icon.

When you return to RPC settings again you will find the RPC url still listed in the same place you originally put it in.

Likewise if you are on Production/mainnet on app that does not take Devnet SOL then

Just letting you know because I have done that by accident on occasion, forgetting to switch my RPC! It caused a few minutes of head-scratching until I realized my mistake.

When should you switch to a custom RPC:

You’re running frequent transactions or tests. The default devnet RPC often gets rate-limited.

  • You’re debugging complex programs. Helius offers enhanced logging and tools.
  • You need precise transaction simulation and debugging. Simulate transactions before they’re sent to avoid costly mistakes on mainnet.
  • You need predictable uptime during a hackathon, workshop, or demo. Public RPCs can go down unexpectedly.
  • You want access to more than just basic features. Custom RPCs unlock features like webhooks, pre-parsed account data, and NFT-specific APIs.
  • You’re building high-volume NFT or token applications. Custom RPCs give you faster indexing, metadata retrieval, and simplified NFT queries.
  • You’re scaling an app or preparing it for production. Higher request limits and predictable performance ensure your app won’t degrade under load.
  • You’re integrating real-time notifications. Webhooks offered by providers like Helius notify your backend instantly when on-chain events occur, eliminating polling and saving resources.

Security warning for devs:

  • Never use the same wallet for Devnet and Mainnet.
  • Never use your main wallet’s private key or seed phrase when testing with RPCs.
  • Don’t share your custom RPC URL with the embedded API key in public projects or GitHub
  • Use environment variables for your API keys. Store your Helius or QuickNode API keys safely in environment variables (.env) instead of hard-coding them into your app to avoid accidental exposure.
  • Use different API keys for different apps. Don’t reuse the same API key across multiple projects.
