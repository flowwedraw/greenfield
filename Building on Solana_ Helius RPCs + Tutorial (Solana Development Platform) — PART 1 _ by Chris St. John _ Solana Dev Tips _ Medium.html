The Basics: What are RPCs?

RPCs are an important building block for Solana devs. We’re going to go into a lot more detail, but this section is to get familiar with terms.

  • “RPC” is a “Remote Procedure Call”, a special type of API protocol devs use to communicate with blockchains like Solana.
  • “RPC nodes” are servers running the RPC protocol. These nodes are an intermediary allowing developers to send requests to the blockchain and receive responses back in a predictable, structured format.
  • “JSON-RPC” is a certain type of RPC protocol used on Solana to communicate with the blockchain.
  • Developers use JSON-RPC to send specific “method” requests to the RPC nodes and blockchain like fetching account balances or placing transactions. Then they receive responses back formatted in JSON. This is required for communication between your app and the Solana blockchain.
  • An RPC “method” is like a function name that performs a specific task. Devs include required and/or optional input variables and receive consistent output formatted responses back.
  • You can use HTTP methods or Websocket. HTTP always uses POST and is best for single requests. Use Websocket for bi-direction subscriptions and real-time events.
  • Solana RPC “endpoints” are similar to a “base url” you may have used with a REST or GraphQL API.
  • Endpoints vary depending on the Solana environment you are using such as devnet (for devs), testnet (QA) and mainnet-beta (the live production Solana blockchain). Also, RPC providers require you to substitute in their RPC provider url if you are using their services.

Example of an HTTP JSON-RPC request (example shows using cURL from CLI):

curl https://api.devnet.solana.com -s -X POST -H "Content-Type: application/json" -d '
  {
    "jsonrpc": "2.0",
    "id": 1,
    "method": "getBalance",
    "params": [
      "83astBRguLMdt2h5U1Tpdq5tjFoJ6noeGwaY3mDLVcri"
    ]
  }
'

# ✅ This is the Solana devnet endpoint: https://api.devnet.solana.com
# ✅ POST method is always used for HTTP calls.
# ✅ Header content type is always JSON.
# ✅ The method can vary, but here it is getBalance
# ✅ Params required: Varies by method, but in this case a public key (pubkey)
# ✅ Params are value only by array order (not key-value pair).

You‘ve probably used an ’“API Gateway” before. It’s a common API management service in cloud platform stacks using API protocols like HTTP REST, WebSockets, or GraphQL.

  • “RPC providers” like Helius are similar to an API Gateway for RPC requests, nodes and methods to connect to the blockchain.
  • The purpose of a Solana RPC provider is similar to the role of providers of cloud API Gateways. They provide a cloud platform to enhance performance, reliability, scalability and provide additional tools for developers. They charge for usage and premium services.

Leading RPC service providers like Helius, Alchemy, QuickNode, and others offer developer-friendly platforms that make working with Solana’s JSON-RPC protocol easier. That’s why you should use those, and not go direct to Solana.

But it’s a bad choice not to, because there are significant limitations that can severely impact your development experience and the performance of your app. Without a provider, you face limits on performance, scaling, additional features, reliability, uptime and cost efficiency.

RPC provider platforms optimize for better connectivity — so you can focus on building great applications. They run the nodes (special servers handling and scaling RPC API requests) that improve performance and offer special features supercharging your Solana requests.

Also, platforms like Helius, Alchemy, Moralis and QuickNode, offer additional tutorials, blog articles, and other API-related tools, including for other blockchains, and may offer resources for entry-level Solana devs.

More on RPC Nodes

As mentioned above, RPC nodes are to Solana what an API Gateway is to many other non-Solana apps and cloud services.

RPC nodes are essential for Solana interaction. They allow apps to read and write data to the Solana blockchain.

As a Solana developer, you would mostly use an RPC provider like Helius to connect to the Solana blockchain for transactions, info queries and interacting with on-chain apps. You take the response and use it in your app.

There are at least two major different types of nodes offered by Helius and most providers.

  1. Shared RPC nodes cater to smaller projects. These are usage-based and rate-limited nodes designed for moderately demanding projects, balancing cost and performance. This would be for a free or low-cost plan.
  2. Dedicated RPC nodes are for high-demand applications. They provide unlimited requests and low latency, ideal for large-scale use cases.

You may also see public/private endpoints referred to.

  • “Public” RPCs would be the native Solana endpoint. These have very low “rate limits” for preventing abusive high usage. Therefore, they aren’t ideal for production apps.
  • “Private” RPCs are what you pay for to get higher rate limits, better performance and reliability. These are what RPC providers offer.

Helius nodes are globally distributed for efficiency: Helius says they have nodes in New York, Singapore, Tokyo, Frankfurt, and other key locations, ensuring low-latency access worldwide.

Nodes automatically route requests. The system sends requests to the nearest node, ensuring optimal performance for shared RPC users.

With Helius “you can lockdown your endpoints to only serve requests for certain IPs and Domains” however, for a client-side app especially, you may need to proxy yoru requests for better security because you do not want to leak your key, as you are paying for usage.

Helius RPC Proxy: There is a free Github 1-click deploy open-source RPC proxy that you can deploy with 1-click to Cloudflare.

Other Related Data Services

Helius also offers additional data services that are there to make life easier for developers, here are a few:

  • Real-time data streaming options. Developers can stream real-time Solana data using tools like Geyser, Enhanced Websockets, and Standard Websockets.
  • A Geyser (“Yellowstone”) plugin that offers the fastest updates on Solana data. “Highly configurable real-time data streams.” This taps into Solana’s data stream, delivering updates on blockchain events such as new blocks being added, transactions being processed, or changes to account states. “Geyser or gRPC streams provide the fastest and most efficient way to stream Solana data directly to your backend. With gRPC, you can subscribe to blocks, slots, transactions, and account updates.”
  • Enhanced Websockets provide transaction and account updates, suited for non-latency-critical client-side applications.
  • Standard Websockets require reconnection handling for two-way communication for real-time data but need robust reconnection logic.
  • Webhooks automate event notifications: Supporting over 70 transaction types, webhooks handle large-scale interactions with up to 100,000 addresses per webhook. They are ideal for automation but not for sub-second latency requirements.

Solana RPC Methods list

These are the methods available. Solana HTTP RPCs

  • getAccountInfo: Retrieve detailed information about a specific account.
  • getBalance: Obtain the balance of a particular account in lamports (the smallest unit of SOL).
  • getBlock: Fetch details of a confirmed block by its slot number.
  • getBlockCommitment: Get commitment information for a specific block.
  • getBlockHeight: Retrieve the current block height of the node.
  • getBlockProduction: Obtain information about block production for the current or a previous epoch.
  • getBlockTime: Fetch the estimated production time of a block.
  • getBlocks: Retrieve a list of confirmed blocks between two slot numbers.
  • getBlocksWithLimit: Get a list of confirmed blocks starting at a given slot, limited by a specified amount.
  • getClusterNodes: Obtain information about all the nodes participating in the cluster.
  • getEpochInfo: Retrieve information about the current epoch.
  • getEpochSchedule: Fetch the epoch schedule, detailing slot and epoch structure.
  • getFeeForMessage: Estimate the fee for a given message.
  • getFirstAvailableBlock: Get the slot of the first available block in the ledger.
  • getGenesisHash: Retrieve the genesis hash of the ledger.
  • getHealth: Check the health status of the node.
  • getHighestSnapshotSlot: Obtain the highest slot for which there is a snapshot.
  • getIdentity: Fetch the identity public key of the current node.
  • getInflationGovernor: Retrieve the current inflation governor parameters.
  • getInflationRate: Get the current inflation rate, current epoch.
  • getInflationReward: Fetch the inflation reward for a list of addresses.
  • getLargestAccounts: Addresses of the largest accounts by lamport balance.
  • getLatestBlockhash: Retrieve the latest blockhash.
  • getLeaderSchedule: Fetch the leader schedule for an epoch.
  • getMaxRetransmitSlot: Get the max slot seen from retransmit.
  • getMaxShredInsertSlot: Retrieve the max slot inserted from shreds.
  • getMinimumBalanceForRentExemption: Calculate the minimum balance needed to make an account exempt from rent.
  • getMultipleAccounts: Info about multiple accounts in a single request.
  • getProgramAccounts: All accounts owned by a specific program.
  • getRecentPerformanceSamples: Obtain a list of recent performance samples.
  • getRecentPrioritizationFees: Fetch recent prioritization fees.
  • getSignatureStatuses: Get the status of a list of signatures.
  • getSignaturesForAddress: Confirmed signatures for transactions involving an address.
  • getSlot: Fetch the current slot the node is processing.
  • getSlotLeader: Obtain the current slot leader.
  • getSlotLeaders: Retrieve the slot leaders for a range of slots.
  • getStakeMinimumDelegation: Get the minimum stake delegation amount.
  • getSupply: Fetch information about the total supply of SOL.
  • getTokenAccountBalance: Retrieve the token balance of a token account.
  • getTokenAccountsByDelegate: Get token accounts associated with a delegate.
  • getTokenAccountsByOwner: Fetch SPL Token accounts owned by a specific account.
  • getTokenLargestAccounts: Retrieve the largest accounts of a specific token type.
  • getTokenSupply: Get the total supply of a specific SPL token.
  • getTransaction: Returns transaction details for a confirmed transaction
  • getTransactionCount: Returns the current Transaction count from the ledger.
  • getVersion: Obtain the current software version of the node.
  • getVoteAccounts: Account info and associated stake for all the voting accounts in the current bank.
  • requestAirdrop: Request airdrop of lamports to a specific address (available only on test networks).
  • sendTransaction: Submit a signed transaction to the network.
  • simulateTransaction: Simulate a transaction to see its effects without broadcasting it.

DAS API Methods

note: this is an API extension apart from core RPC methods, some may not work with some providers.

  • getAsset: Metadata information of a compressed/standard asset
  • getAssets: Metadata for multiple assets in a single request
  • getAssetProof: Merkle proof for a specific asset
  • getAssetProofs: Merkle proofs for multiple assets
  • getAssetsByAuthority: Assets controlled by a specific authority
  • getAssetsByCreator: Assets created by a specified creator
  • getAssetsByGroup: Assets using custom group identifiers, i.e. collection
  • getAssetsByOwner: Assets owned by a specific wallet address
  • getAssetSignatures: Transactions signatures for a specified compressed asset
  • getTokenAccounts: List token accounts and balances for a specified mintAddress or ownerAddress
  • getNftEditions: Details of edition NFTs of a specified master NFT
  • searchAssets: Search for assets matching specific parameters

We will be delving into these in more detail. Some may not make sense to you yet, and many are only needed in specialized cases. All the metadata offered in that result above can be used for learning more about the token.

  • Postman makes it easy to craft and test Solana RPC requests.
  • No need to set up a full dev environment for basic method testing to get basic info about the request and results.
  • Use and build collections (like we did) of reusable requests for common Solana RPC methods. Add more like with the DAS API.
  • Share and check in to Github customized collections or environments.
  • Quickly test individual RPC calls and inspect responses for errors or unexpected data.
  • Keep a request/response history/logs automatically.
  • Automate Solana RPC workflows using Postman’s pre-request scripts and tests.
  • Easily adjust environment, parameters and headers in the Postman interface to troubleshoot requests.
  • Save and organize requests for repeatable debugging workflows.

Typescript version

It’s awesome we got it working, but best practice is to get this in TypeScript.

Let’s rewrite the code for typescript.

Install typescript libraries

npm install --save-dev typescript @types/node

The latest version of axios should include types. I am using “axios”: “^1.7.9”

Install ts-node

npm install --save-dev ts-node

Initialize a tsconfig.json file:

npx tsc --init

.tsconfig file should be like this:

{
  "compilerOptions": {
    "target": "ES6",
    "module": "CommonJS",
    "outDir": "./dist",
    "strict": true,
    "esModuleInterop": true,
    "resolveJsonModule": true,
    "skipLibCheck": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules"]
}

Create src directory

mkdir src
cd src

Make a new typescript file:

import axios, { AxiosRequestConfig }  from 'axios';

const API_KEY = "[your key]"; // Replace with your actual API key
const URL = `https://mainnet.helius-rpc.com/?api-key=${API_KEY}`;

const data = {
  jsonrpc: "2.0",
  method: "getAsset",
  params: [
    "WENWENvqqNya429ubCdR81ZmD69brwQaaBYY6p3LCpk"
  ],
  id: 0
};

const config: AxiosRequestConfig = {
  method: 'post',
  maxBodyLength: Infinity,
  url: URL,
  headers: {
    'Content-Type': 'application/json'
  },
  data: data
};

async function getAsset() {
  try {
    const response = await axios.request(config);
    console.log("Response Data:\n", JSON.stringify(response.data, null, 2));
  } catch (error: any) {
    if (error.response) {
      console.log("Error Response:\n", JSON.stringify(error.response.data, null, 2));
    } else {
      console.log("Error:\n", error.message);
    }
  }
}

getAsset();
